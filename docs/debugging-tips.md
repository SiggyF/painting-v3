# Debugging Tips: The Case of the Faint Blue Patches

## Problem Description
Upon selecting a simulation model, faint blue patches would appear in the water areas. These patches were visible on the screen but disappeared when the simulation was cleared. The goal was to identify which texture or stage of the pipeline was introducing this artifact.

## Diagnostic Steps

### 1. Automated Texture Readback
We implemented a `countBluePixels` function to scan GPU textures during the simulation loop.
- **Goal**: Determine if "blue" was stored as physical pigment data.
- **Logic**: Used `device.queue.copyTextureToBuffer` to pull data from `Main`, `Source`, and `Paint` textures into staging buffers.
- **Threshold**: Scanned for pixels where `B > R` and `B > G` with a very low sensitivity threshold (`> 0.001`).
- **Result**: All simulation textures returned **0** blue pixels, despite the patches being visible on screen.

### 2. Manual Pixel Sampling (Alt-Click)
We implemented an "Alt-Click" feature to sample the exact RGBA values of a specific pixel across all layers.
- **DOM Canvases**: Used `ctx.getImageData` to sample 2D canvases.
- **WebGPU Canvas**: Used a persistent `outputTex` and `copyTextureToTexture` to capture the final display frame before it was cleared by the swap chain.
- **Result**:
    - `WebGPU Main/Source/Paint`: All reported `[0.0000, 0.0000, 0.0000, 0.0000]`.
    - `WebGPU Display Canvas`: Reported non-zero blue-dominant values (e.g., `R=8 G=12 B=21`).

### 3. Conclusion: Procedural Noise
The discrepancy between the empty simulation textures and the blue-dominant display output proved that the artifact was **procedural**. It was not stored in memory but calculated on-the-fly in the fragment shader.

The culprit was the "Background Flow Visualization" in the `render_main` shader:
```wgsl
let streakColor = vec3<f32>(0.4, 0.6, 1.0); // Blue-dominant color
let streakNoise = noise(uvA * 20.0 + params.time * 0.1);
var streakAlpha = flowLum * 0.15; // 15% opacity noise
```
This logic was active whenever `params.analytical` was `0.0` (which occurs when a numerical model is selected).

## Lessons Learned
1. **Trust the Readback**: If `copyTextureToBuffer` reports zeros, the data is not in the texture.
2. **Check the Final Pass**: Visual artifacts that don't appear in simulation data are almost always introduced in the final `render` fragment shader.
3. **Swap Chain Ephemerality**: You cannot reliably sample the current WebGPU display texture (`context.getCurrentTexture()`) inside an event handler, as the browser clears it immediately after the frame. Use a persistent intermediate texture (`copyTextureToTexture`) for reliable debugging.
4. **Procedural vs. Data**: Always distinguish between artifacts stored in state (pigment) and those generated by logic (noise/visualization).
